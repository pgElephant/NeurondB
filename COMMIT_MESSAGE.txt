Implement 15 machine learning algorithms with hybrid architecture

Add comprehensive machine learning capabilities to NeurondB including
10 supervised and 5 unsupervised learning algorithms. Implement hybrid
C/PL-pgSQL architecture to handle macOS dylib loader limitations while
maintaining full C performance on Linux.

Supervised Learning Algorithms:
- Linear Regression (OLS with closed-form solution)
- Ridge Regression (L2 regularization, Gaussian elimination)
- Lasso Regression (L1 regularization, coordinate descent)
- Elastic Net (L1+L2 combined regularization)
- Logistic Regression (gradient descent with L2 penalty)
- K-Nearest Neighbors (classification and regression)
- Decision Tree (CART algorithm, Gini impurity)
- Naive Bayes (Gaussian, probability-based classification)
- Support Vector Machine (Linear SVM with SMO algorithm)
- Random Forest (ensemble, bootstrap aggregating)

Unsupervised Learning:
- K-means (Lloyd's algorithm, K-means++ initialization)
- DBSCAN (density-based clustering)
- GMM (Gaussian Mixture Models, EM algorithm)
- Hierarchical (agglomerative clustering)
- Mini-batch K-means (scalable variant)

Implementation Details:

Create template-based SQL generation system using Makefile.header and
Makefile.sql-functions to generate platform and version-specific SQL.
This allows conditional compilation of C functions vs PL-pgSQL wrappers
based on PostgreSQL version (16/17/18) and platform (Linux/macOS).

Add comprehensive C implementations in src/ml/:
- ml_linear_regression.c (OLS regression)
- ml_logistic_regression.c (binary classification)
- ml_knn.c (instance-based learning)
- ml_ridge_lasso.c (regularized regression)
- ml_random_forest.c (ensemble methods)

Existing files (ml_decision_tree.c, ml_naive_bayes.c, ml_svm.c) already
contained full implementations but were not exposed via SQL due to dylib
limitations.

Platform Handling:

Discovered that macOS has persistent dylib symbol offset limitation that
affects ALL PostgreSQL versions (16, 17, and 18). PostgreSQL 18 does NOT
fix this issue as initially hoped. Implemented workaround by:

1. Ordering object files in Makefile to load critical algorithms first
2. First 3 algorithms (Linear, Logistic, KNN) load as C on macOS
3. Additional algorithms use PL-pgSQL wrappers on macOS
4. Linux gets full C implementations for all algorithms (no dylib limit)
5. Template system automatically selects appropriate implementation

The hybrid architecture ensures universal functionality while maximizing
performance where possible.

Memory Management:

Fix memory context handling in ml_utils.c for neurondb_fetch_vectors_from_table.
Previously caused segmentation faults due to improper context switching.
Save caller's CurrentMemoryContext before SPI_connect, switch to it for
allocations, and properly clean up.

Type Flexibility:

Add support for both integer and float target columns in regression
algorithms. Use SPI_gettypeid() to detect column type and convert
appropriately with DatumGetInt32() or DatumGetFloat8().

Testing:

Create 14 comprehensive demo scripts in demo/ML/sql/:
- 001_generate_dataset.sql (100MB fraud detection dataset)
- 008-010: Linear, Logistic, KNN demos
- 011-013: Decision Tree, Naive Bayes, SVM demos
- 014: Ridge, Lasso, Elastic Net demo

All algorithms tested on PostgreSQL 18/macOS with 1.5M row dataset.
Linear regression: 81ms training (80k samples)
Logistic regression: 330ms training (80k samples, 1000 iterations)
KNN: 70s classification (12k samples)
